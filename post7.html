CROSS-SITE SCRIPTING

Cross-site scripting (XSS) is a type of cyber-attack where the attacker injects malicious codes into a legitimate web page or application. It is a client-side code injection attack, so the user is the one who is at risk. The attack occurs when the user visits the infected web page or application; the web page acts as a medium to transport the malicious code to the user’s browser. Websites that allow users to leave comments or reviews are more vulnerable to XXS attacks.

The attack may be made to trick the user into exposing their data. Also, defaming a website can be achieved by modifying the content on the website or redirecting the user to another website.

How cross-site scripting works?

When attackers insert their code into a web page, which is usually done by exploiting a bug in the software of the website, they can then insert their script which is executed by the browser of the victim.

Since the JavaScript runs on the victim’s browser tab, the session that stolen confidential information about the authenticated user, allows the intruder to attack the site administrators.

Attackers may use cross-site scripting in several ways, depending on their objective. Some of the common types are listed below: 1. STORED (persistent) CROSS-SITE SCRIPTING:

It is one of the most dangerous and commonly used types of XSS attacks. When the attackers come across a vulnerable web page or application, they inject the malicious code into its server. Then whenever a user visits the webpage, the malicious code executes in the user's browser. Now the attacker can steal user’s personal information associated with the website. This type of attack is common on platforms that allow users to share their content, messages, or videos.

2. REFLECTED (non-persistent) CROSS-SITE SCRIPTING:

In reflected cross-site scripting, the attacker sends the malicious code via a link. When the user clicks the link it gets executed in the user’s browser. These malicious links are easy to include in emails. Then, the browser transfers the entire user’s private information to the attacker.

3. SELF CROSS-SITE SCRIPTING:

Self cross-site scripting happens when attackers manipulate a flaw that requires incredibly precise context and manual modifications. The last one who can be a suspect is himself.

4. BLIND CROSS-SITE SCRIPTING:

Blind cross-site scripting arises when an attacker cannot see the outcome of an attack. Generally, the vulnerabilities lie on the page where only the administrator can visit therefore the execution of the payload can take a longer period. If the payload crashes, the attacker won’t be notified.

5. DOM-BASED CROSS-SITE SCRIPTING:

DOM-based cross-site scripting is hard to track from the server-side, as usually malicious payloads do not enter the host. The source of the problem is in the client-side code. Avoid using network-received data for sensitive actions such as editing or redirecting on the client-side.

Prevention from cross-site scripting attacks:

1) Anyone involved in the development of a web application must be aware of the consequences of cross-site scripting vulnerabilities.

2) Do not trust any user input, even though they have signed in and authenticated their accounts. Any input from the client-side should be checked, filtered, and encoded.

3) Scan your web applications using a web vulnerability scanner regularly. It can be possible that XSS vulnerabilities are introduced from the developer side or through external libraries.

4) You can practically patch attacks against your website using a firewall. This approach intercepts attacks long before malicious requests hit your website ever. This also has the advantage of defending against such large-scale attacks as DDOS.

5) To minimize the effects of a possible XSS vulnerability, set the HttpOnly flag for cookies. If you do, those cookies will not be accessible through JavaScript on the client-side.
